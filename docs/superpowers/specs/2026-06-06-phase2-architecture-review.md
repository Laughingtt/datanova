## 2. Code Alignment Issues

### 🔴 C1: AgentTool signature mismatch (inherited from P1)

Same as Phase 1 C1. All three new tools — `lookup_semantic_layer`, `lookup_examples`, `ai_suggest_semantic_layer` — use the wrong `execute: async (args) => ...` signature.

Additionally, `ai_suggest_semantic_layer` (Task 3 Step 3) returns:
```typescript
return {
  needs_semantic_suggestion: true,
  tables: tablesSummary,
  instruction: "..."
};
```

This suffers the same problem as P1's `ai_annotate_schema`: it returns a prompt asking the LLM to generate recommendations, but there's no tool for the Agent to save them. The plan says "call `save_semantic_suggestions` action" — this tool doesn't exist.

**Fix**: Same as P1 — rewrite all tools to match `(_toolCallId, params) => { content: [...], details: {} }`. For `ai_suggest_semantic_layer`, either (a) call LLM directly inside the tool and save as drafts, or (b) create a `save_semantic_suggestions` tool.

### 🟡 C2: `testMetric` in store.ts violates layer separation

Plan Task 1 Step 4 adds `testMetric()` to `store.ts`:
```typescript
export async function testMetric(datasourceId: string, metricId: string): Promise<QueryResult> {
  const metric = getMetric(metricId);
  // ... builds SQL ...
  return executeSql(datasourceId, sql, { timeout: 5000, rowLimit: 10 });
}
```

This imports `executeSql` from `mysql/executor.js` into `store.ts`. Currently, `store.ts` only imports from `crypto` and `better-sqlite3` — it has no MySQL dependencies. Adding this would couple the SQLite store layer to the MySQL execution layer.

**Fix**: Move `testMetric` to a new file like `packages/server/src/agent/semantic-test.ts` or into `routes/semantic.ts` as a route handler helper. Keep store.ts as pure SQLite CRUD.

### 🟡 C3: Semantic layer SQL generation is entirely prompt-based — no code

The spec requirement "Deterministic SQL generation from semantic layer" is implemented ONLY via system prompt instructions:

> "If a metric is found, generate SQL by combining: 1. The metric's sql_expression in SELECT, 2. dimension sql_expressions from the model in GROUP BY, ..."

There is **no code** that actually builds the SQL from metric + dimension + model data. The LLM is expected to do this by following the prompt. This defeats the purpose of the semantic layer — the whole point is that metric-based queries should be **deterministically** generated (100% accuracy), not probabilistically generated by the LLM.

**Why this matters**: If the LLM misinterprets the semantic layer data, the same metric query could produce different SQL on different runs. The spec explicitly says "语义层命中的查询准确率接近100%", but a prompt-only approach cannot guarantee that.

**Fix**: Create a `buildSemanticSql()` function in a new file like `packages/server/src/agent/semantic-sql-builder.ts`:

```typescript
export function buildSemanticSql(options: {
  metric: Pick<SemanticMetric, 'sql_expression' | 'filters' | 'name'>;
  dimensions: Array<Pick<SemanticDimension, 'sql_expression' | 'name'>>;
  model: Pick<SemanticModel, 'base_table' | 'joins'>;
  userFilters?: Array<{ column: string; operator: string; value: string }>;
}): string {
  // Deterministically build SQL from metric + dimensions + model
  const selectParts = [`${metric.sql_expression} AS ${metric.name}`];
  const groupByParts: string[] = [];
  
  for (const dim of dimensions) {
    selectParts.push(`${dim.sql_expression} AS ${dim.name}`);
    groupByParts.push(dim.sql_expression);
  }
  
  let sql = `SELECT ${selectParts.join(', ')} FROM ${model.base_table}`;
  
  const joins = JSON.parse(model.joins) as Array<{table: string; on: string; type: string}>;
  for (const j of joins) {
    sql += ` ${j.type.toUpperCase()} JOIN ${j.table} ON ${j.on}`;
  }
  
  const whereParts: string[] = [];
  const filters = JSON.parse(metric.filters) as Array<{column: string; operator: string; value: string}>;
  for (const f of filters) {
    whereParts.push(`${f.column} ${f.operator} '${f.value}'`);
  }
  // Add user filters
  if (options.userFilters) {
    for (const f of options.userFilters) {
      whereParts.push(`${f.column} ${f.operator} '${f.value}'`);
    }
  }
  
  if (whereParts.length > 0) {
    sql += ` WHERE ${whereParts.join(' AND ')}`;
  }
  
  if (groupByParts.length > 0) {
    sql += ` GROUP BY ${groupByParts.join(', ')}`;
  }
  
  return `/* source: semantic_layer */ ${sql}`;
}
```

This function should be called inside the `lookup_semantic_layer` tool's return value, so the Agent receives the pre-built SQL and just needs to execute it, not generate it. The system prompt should say: "If the lookup returns a `generated_sql` field, execute it directly without modification."

## 3. Architecture Risks

### ⚠️ R1: `lookup_semantic_layer` keyword matching is fragile

The tool uses `includes()` for matching:
```typescript
m.name.toLowerCase().includes(queryLower) ||
m.display_name.toLowerCase().includes(queryLower)
```

This means searching "销售" would match "商品交易总额" if it contained "销售" as a substring, but would miss "GMV" → "商品交易总额" because "GMV" doesn't appear in the name/display_name.

The alias matching helps:
```typescript
const aliasMatch = JSON.parse(m.aliases).some(a => a.toLowerCase().includes(queryLower));
```

But for Chinese text, substring matching is weak. "华东区销售额" won't match metric "gmv" with alias "销售额" because the full query includes "华东区" — the `includes()` check for "华东区销售额" against alias "销售额" would actually work (since "销售额" is a substring of "华东区销售额"). But partial matches could be improved.

**Recommendation**: Split the query into keywords (split by spaces, Chinese characters) and check if ANY keyword matches. Also, consider checking if the metric's `dimensions` array contains the query term (e.g., "华东区" might match a "region" dimension's values).

---

### ⚠️ R2: `lookup_examples` scoring is simplistic

The plan uses keyword overlap scoring:
```typescript
for (const kw of keywords) {
  if (questionLower.includes(kw)) score += 2;
  for (const t of tablesUsed) {
    if (t.toLowerCase().includes(kw)) score += 1;
  }
}
```

For Chinese queries, splitting by whitespace is wrong: "华东区销售额" is one token with no spaces. The keyword extraction should use a simple Chinese segmentation approach (at minimum, split by common delimiters and extract table names).

**Recommendation**: Use a simple multi-strategy matching: (1) exact substring match, (2) table name match, (3) extract numbers and match against SQL patterns. Don't try to do proper Chinese segmentation in Phase 2 — just improve the naive split.

---

### ⚠️ R3: Auto-save query examples may accumulate garbage

Phase 2 Task 5 Step 1 auto-saves every successful query to `query_examples`. But "successful" just means "executed without error and returned rows" — it doesn't mean the query was correct. A query that returns wrong data (e.g., wrong table, wrong filter) will still be auto-saved and later used as a Few-Shot example.

The plan partially addresses this with the feedback loop (positive feedback → verified), but there's no mechanism to retroactively flag auto-saved examples that were later found to be wrong.

**Recommendation**: Auto-saved examples should start with `is_verified: 0` (the plan does this) and a `confidence_score` field. Examples should only be injected as Few-Shot if `is_verified: 1` OR `success_count >= 3`. This prevents one-off accidental successes from polluting the Few-Shot pool.

---

### ⚠️ R4: Semantic layer data model — `filters` as JSON text is fragile

The `semantic_metrics.filters` field stores JSON like:
```json
[{"column":"orders.status","operator":"=","value":"completed"}]
```

This requires parsing JSON every time the metric is used. More importantly, the `buildSemanticSql()` function (if implemented as recommended in C3) would need to interpolate these values directly into SQL strings, which is a SQL injection risk if the values come from user input.

The fix in C3 uses string interpolation (`'${f.value}'`), but the filter values should come from the metric definition (admin-curated), not user input. User-specified filters should be parameterized.

**Recommendation**: In `buildSemanticSql()`, metric-defined filters can use interpolation (they're admin-curated). User-specified filters should be handled differently — pass them as a separate parameter and validate/sanitize them before inclusion.

---

### ⚠️ R5: MetricsPage UI is large — should be broken into smaller tasks

Task 4 (Frontend Metrics Management Page) creates 7 new components and modifies 4 files. This is the largest single task in Phase 2. It should be broken down:

- Task 4a: MetricForm + DimensionForm + ModelForm (data entry)
- Task 4b: MetricsPage layout + list view
- Task 4c: AISuggestPanel + test metric feature
- Task 4d: QueryExamplesSection integration

## 5. Missing Items

1. **`buildSemanticSql()` function** — The most important missing piece. Without deterministic SQL generation, the semantic layer loses its core value proposition (100% accuracy for metric queries). See C3 for the proposed implementation.

2. **`save_semantic_suggestions` tool** — Same issue as P1's `save_annotations` tool. The `ai_suggest_semantic_layer` tool asks the Agent to call this action, but it doesn't exist.

3. **`generateAnnotationSkill()` extension** — After adding metrics/dimensions, the auto-generated SKILL.md should include semantic layer info. Neither Phase 1 nor Phase 2 addresses this.

4. **`refreshHarnessSkills()` on metric changes** — When a metric is created/updated/deleted, active AgentHarness instances need their skills refreshed. The plan doesn't include this.

5. **Migration versioning** — Phase 2 adds 5 new tables. No migration framework. Should add `schema_version` tracking in `app_config` as recommended in P1 review.

6. **`conversationId` in feedback API** — Phase 2 Task 5 Step 2 references `findQueryExampleByMessageId(msgId)` — this function doesn't exist in the plan. It needs to query `query_examples` by matching the `question` field to the message's original question.

## 6. Recommendations (Priority Order)

### Must Fix Before Implementation

1. **Implement `buildSemanticSql()` function** — This is the #1 priority. Without it, the semantic layer is just a search index, not a deterministic SQL generator. The whole accuracy promise (72%→90%) depends on this.
2. **Fix AgentTool signatures** — Same as P1 C1.
3. **Fix `ai_suggest_semantic_layer` tool** — Either make it self-contained (calls LLM internally) or add a `save_semantic_suggestions` tool.
4. **Move `testMetric` out of store.ts** — Keep store.ts as pure SQLite CRUD.

### Should Fix Before Implementation

5. **Improve keyword matching** in `lookup_semantic_layer` — at minimum, split Chinese text and check aliases/dimensions.
6. **Add `confidence_score` or `min_success_count` filter** to Few-Shot example injection — don't inject unverified examples.
7. **Extend `generateAnnotationSkill()`** to include semantic layer metrics/dimensions.
8. **Break Task 4 into 4 sub-tasks** for the Metrics Management UI.

### Can Fix During Implementation

9. **Continue with Zustand view switching** — no React Router needed yet.
10. **Add `schema_version` to `app_config`** for migration tracking.
11. **Parameterize user-specified filters** in `buildSemanticSql()`.

---

## 7. Verdict

**Phase 2 is implementable with one critical addition: `buildSemanticSql()`.** Without deterministic SQL generation from the semantic layer, Phase 2's core value proposition collapses — it would be a search index plus prompt engineering, not a semantic layer. The plan's data model is sound, the API design is consistent, and the UI scope is reasonable. But the semantic layer's promise of "100% accuracy for metric queries" cannot be delivered through prompt instructions alone.

**Estimated effort adjustment**: +2-3 days for `buildSemanticSql()` + integration with `lookup_semantic_layer` tool. This is the single most impactful code addition in the entire 4-Phase project.
