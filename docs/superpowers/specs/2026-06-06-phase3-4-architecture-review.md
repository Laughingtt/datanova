## Phase 3: Intelligent Attribution + Semantic Layer Closure

### Executive Summary

Phase 3 is the smallest phase (2 tasks). The attribution analysis is implemented primarily through system prompt instructions, which is the right approach — the Agent already has the tools (execute_sql, discover_schema) to perform decomposition queries. The frontend AttributionView component is well-designed. The main concern is that attribution effectiveness **depends critically on Phase 2's `buildSemanticSql()` function** being implemented — without it, the Agent cannot reliably generate the multiple decomposition queries needed.

### Issues

#### 🟡 C1: Attribution relies on multiple sequential queries — no orchestration code

The attribution flow requires:
1. Confirm fact → 1 query
2. Decompose by dimension 1 → 1 query
3. Decompose by dimension 2 → 1 query
4. Decompose by dimension 3 → 1 query
5. Cross-reference → 1 query
6. Generate conclusion → 0 queries

That's 5+ sequential queries, all generated by the LLM following system prompt instructions. If any one query is wrong, the entire attribution chain fails. This is acceptable for Phase 3 (MVP), but should be noted as a limitation.

**Recommendation**: If Phase 2 implements `buildSemanticSql()`, the decomposition queries can be generated deterministically (swap GROUP BY dimension), making attribution much more reliable. Document this dependency clearly.

#### 🟡 C2: `AttributionView` contribution chart is basic

The plan implements a horizontal bar chart using pure CSS divs (no chart library). This works for simple cases but won't handle:
- Negative + positive contributions in the same chart (waterfall)
- Many dimension values (needs scrolling/truncation)
- Interactive hover/click to drill into a specific factor

**Recommendation**: For Phase 3, the CSS-based chart is fine. If a chart library (e.g., recharts, which is React-friendly) is already in the project's consideration, use it. Otherwise, keep it simple and note as a Phase 4+ enhancement.

#### 🟢 C3: Metric version management is minimal but sufficient

Task 2 adds `version INTEGER NOT NULL DEFAULT 1` and auto-increments on update. This is sufficient for basic version tracking. No version history table (which would store each version's SQL expression for audit). Acceptable for Phase 3.

### Phase 3 → Phase 4 Dependency Check

|---|---|---|

**No blocking dependencies.** Phase 3 is self-contained.

### Verdict

**Phase 3 is implementable as-is.** It's small, well-scoped, and the prompt-based approach is appropriate for MVP attribution. The main risk is attribution reliability without `buildSemanticSql()` from Phase 2 — document this dependency.

## Phase 4: Smart Reports + Scheduled Queries + Data Dictionary

### Executive Summary

Phase 4 adds three independent subsystems. The scheduled queries subsystem is the most complex (cron execution, alert conditions, persistence). The data dictionary is the simplest (read-only search layer). The smart reports are prompt-driven (like attribution). The main issues are: (1) `node-cron` process lifecycle management, (2) alert notification delivery mechanism (WebSocket broadcast from scheduler), (3) execution history storage (only last_run_result, no history table), and (4) the report template system is under-specified.

### Issues

#### 🔴 C1: `node-cron` lifecycle management — scheduler.ts has critical bugs

**Problem 1: `require()` in ESM context**

Plan code (Task 1 Step 4):
```typescript
export function startScheduler(): void {
  const { listDatasources } = require("./store.js") as { listDatasources: () => any[] };
```

The project uses ESM (`"type": "module"` in package.json). `require()` doesn't work in ESM. Must use `import`.

**Problem 2: No graceful shutdown coordination**

The plan adds `startScheduler()` and `stopScheduler()` to index.ts. But `stopScheduler()` only stops cron tasks — it doesn't wait for in-flight executions to complete. If a scheduled query is mid-execution when SIGTERM arrives, it could:
- Leave `last_run_status` as null (never updated)
- Leave the MySQL connection hanging

**Problem 3: `node-cron` doesn't persist state**

If the server crashes (not graceful shutdown), all cron tasks are lost until restart. The plan handles this by re-registering on startup, which is correct. But there's no mechanism to detect missed executions (e.g., a daily report that should have run at 9am but the server was down from 8:55 to 9:10).

**Fix**:
1. Replace `require()` with `import`
2. Add `executeScheduledQuery` abort controller support for graceful shutdown
3. Add a `last_expected_run` field to detect missed executions on startup
4. Consider using `setTimeout`-based scheduling instead of `node-cron` for simpler lifecycle management, or accept the limitations

---

#### 🟡 C2: Alert notification delivery — no WebSocket broadcast mechanism

The scheduler runs server-side. When an alert triggers, the plan says "store an alert record in `query_alerts` table" but doesn't push it to the frontend. The spec requires: "alert SHALL appear as a system message in the active conversation."

This requires a WebSocket broadcast from the scheduler to all connected clients. Currently, WebSocket connections are managed per-conversation in `chat-handler.ts`. There's no global broadcast channel.

**Fix options**:
- **Option A**: Maintain a set of all active WebSocket connections globally. When an alert fires, broadcast to all.
- **Option B**: Create a dedicated "System Alerts" conversation that receives alert messages. Users see it in their conversation list.
- **Option C**: Polling — frontend periodically calls `GET /api/datasources/:dsId/query-alerts?since=<lastCheck>`. Simpler but not real-time.

**Recommendation**: Option C for Phase 4 MVP. Add a small polling interval (30s) in the frontend. Option A can be added later for real-time alerts.

---

#### 🟡 C3: Execution history — only `last_run_result` stored

The `scheduled_queries` table only has `last_run_at`, `last_run_status`, `last_run_result` — one entry. The spec requires "view execution history" which implies multiple past executions.

The plan acknowledges this: "Since we only store `last_run_result` (one entry), we need a separate `query_execution_history` table for full history. For now, show the single `last_run_result`."

**Fix**: Add a `query_execution_history` table:
```sql
CREATE TABLE IF NOT EXISTS query_execution_history (
  id TEXT PRIMARY KEY,
  scheduled_query_id TEXT NOT NULL,
  executed_at TEXT NOT NULL,
  status TEXT NOT NULL CHECK(status IN ('success', 'error')),
  result_summary TEXT,
  execution_time_ms INTEGER,
  row_count INTEGER,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (scheduled_query_id) REFERENCES scheduled_queries(id) ON DELETE CASCADE
);
```

Insert a row after each execution. The `scheduled_queries.last_run_*` fields become denormalized shortcuts for the latest entry.

#### 🟡 C4: Report template customization is under-specified

Task 5 stores templates as JSON in `app_config` with key `report_template_<id>`. But the plan doesn't define:
- What the template JSON schema looks like
- How the Agent uses the template (does it receive the template in the system prompt?)
- How template sections map to Agent queries
- Whether templates can be shared across datasources

**Recommendation**: For Phase 4 MVP, skip custom templates. Just use the default 6-section report structure defined in the system prompt (Task 3 Step 1). Add template customization as a Phase 5 item. The current Task 5 can be deferred.

---

#### 🟡 C5: Data dictionary search is annotation-only — misses schema without annotations

The dictionary search (Task 4 Step 1) searches:
- Metrics (name, display_name, aliases)
- Dimensions (name, display_name, values)
- Tables/columns **only via annotations**

If a table has no annotations, it won't appear in dictionary search results. The spec says the dictionary should search "database tables (name, comment, business description annotation)" — the table name itself should be searchable even without annotations.

**Fix**: Add a direct schema search fallback. When the dictionary search finds no annotation matches for tables, also search `INFORMATION_SCHEMA` for table/column names matching the query. Or cache the schema in SQLite and search it directly.

---

#### 🟢 C6: `ReportExport` HTML export is basic

The HTML export (Task 3 Step 5) generates a simple HTML file with inline CSS. It doesn't include:
- Embedded chart images (if charts are added later)
- Proper encoding for Chinese characters
- Print-friendly layout

Acceptable for Phase 4 MVP. Can be enhanced later.

### Verdict

**Phase 4 is implementable with fixes to scheduler.ts.** The `require()` in ESM context is a blocking issue. The alert notification delivery and execution history are gaps that should be addressed (polling for alerts, add history table). Report template customization should be deferred to keep Phase 4 scope manageable. The data dictionary is the cleanest subsystem — just needs the annotation-less search fix.
