## ADDED Requirements

### Requirement: SQL write-safety enforcement
The `execute_sql` tool SHALL enforce that only SELECT, SHOW, DESCRIBE, and EXPLAIN queries are executed. The existing `isSelectQuery()` function SHALL be called before executing any SQL. If the check fails, the tool SHALL return an error message to the Agent indicating that only read queries are allowed, and the SQL SHALL NOT be sent to MySQL.

#### Scenario: SELECT query allowed
- **WHEN** the Agent calls `execute_sql` with `SELECT * FROM orders LIMIT 10`
- **THEN** the system SHALL pass the `isSelectQuery()` check and execute the query

#### Scenario: DROP TABLE query rejected
- **WHEN** the Agent calls `execute_sql` with `DROP TABLE orders`
- **THEN** the system SHALL reject the query with error "Only SELECT, SHOW, DESCRIBE, EXPLAIN queries are allowed. DROP is not permitted."

#### Scenario: UPDATE query rejected
- **WHEN** the Agent calls `execute_sql` with `UPDATE orders SET status = 'cancelled'`
- **THEN** the system SHALL reject the query with error "Only SELECT, SHOW, DESCRIBE, EXPLAIN queries are allowed. UPDATE is not permitted."

### Requirement: Table and column name validation
The `execute_sql` tool SHALL validate that all table names and column names referenced in the SQL exist in the previously discovered schema for the datasource. If a table or column name does not exist, the tool SHALL return an error listing the invalid names and suggesting valid alternatives (by Levenshtein distance ≤ 2).

#### Scenario: Invalid table name detected
- **WHEN** the Agent calls `execute_sql` with `SELECT * FROM ordr LIMIT 10` and the schema has table `orders`
- **THEN** the system SHALL return error "Table 'ordr' does not exist. Did you mean 'orders'?" and NOT execute the query

#### Scenario: Invalid column name detected
- **WHEN** the Agent calls `execute_sql` with `SELECT amont FROM orders` and the schema has column `amount`
- **THEN** the system SHALL return error "Column 'amont' does not exist in table 'orders'. Did you mean 'amount'?" and NOT execute the query

#### Scenario: No prior schema discovery
- **WHEN** the Agent calls `execute_sql` without having called `discover_schema` first for this datasource
- **THEN** the system SHALL skip table/column validation and proceed with execution (schema may not be cached yet)

### Requirement: Large table WHERE clause check
The `execute_sql` tool SHALL check if the target table has more than 100,000 rows (based on `TABLE_ROWS` from `INFORMATION_SCHEMA.TABLES`) and the SQL does not contain a WHERE clause. If a large table is queried without WHERE, the tool SHALL return a warning to the Agent suggesting to add filtering conditions.

#### Scenario: Large table without WHERE clause
- **WHEN** the Agent calls `execute_sql` with `SELECT * FROM orders` and `orders` has 500,000 rows
- **THEN** the system SHALL return a warning "Table 'orders' has ~500,000 rows. Query without WHERE clause may be slow. Consider adding filtering conditions." and still execute the query (with the default LIMIT 1000)

#### Scenario: Small table without WHERE clause
- **WHEN** the Agent calls `execute_sql` with `SELECT * FROM categories` and `categories` has 50 rows
- **THEN** the system SHALL execute the query without warning

### Requirement: Probe execution before full query
The `execute_sql` tool SHALL support a `probe` mode that executes the SQL with `LIMIT 10` first to check result reasonableness before executing the full query. Probe execution SHALL be automatically triggered for queries generated by NL→SQL (not for semantic-layer queries). The probe checks: (1) result is not empty, (2) result columns match expected columns from the SQL, (3) no unexpected NULL values in key columns. If probe fails, the tool SHALL return the probe result with a suggestion to the Agent to correct the SQL.

#### Scenario: Probe detects empty result
- **WHEN** a NL→SQL generated query returns 0 rows in the probe execution
- **THEN** the system SHALL return the probe result with message "Query returned 0 rows. The filtering conditions may be too restrictive. Consider: (1) checking date ranges, (2) relaxing filters, (3) verifying table relationships." and NOT execute the full query

#### Scenario: Probe detects success
- **WHEN** a NL→SQL generated query returns rows in the probe execution
- **THEN** the system SHALL proceed to execute the full query (with LIMIT 1000)

#### Scenario: Semantic layer query skips probe
- **WHEN** a query is generated from the semantic layer (deterministic mapping)
- **THEN** the system SHALL skip probe execution and directly execute the full query

### Requirement: Empty result auto-correction
When a SQL query returns 0 rows, the Agent SHALL be instructed (via system prompt) to automatically attempt correction. The Agent SHALL analyze the possible causes (wrong table, wrong filter, wrong JOIN) and regenerate the SQL. The system SHALL limit auto-correction attempts to 2 retries. If all retries fail, the Agent SHALL report the issue to the user with suggestions.

#### Scenario: Auto-correction succeeds on first retry
- **WHEN** a query returns 0 rows and the Agent corrects the SQL (e.g., changes date filter from '2025-05' to '2025-04')
- **THEN** the Agent SHALL execute the corrected SQL and return results to the user

#### Scenario: Auto-correction exhausted
- **WHEN** a query returns 0 rows after 2 retry attempts
- **THEN** the Agent SHALL report to the user: "I couldn't find data matching your criteria after 2 attempts. Possible reasons: [list]. Could you provide more details?"

### Requirement: SQL validation status in frontend
The frontend SHALL display SQL validation status for each query execution. When a SQL validation warning or error occurs, the `ChatMessage` SHALL include a `validationStatus` field with `level` (`"error"`, `"warning"`, `"info"`) and `message`. The frontend SHALL render validation warnings as a yellow banner above the result table and validation errors as a red banner.

#### Scenario: Validation warning displayed
- **WHEN** a SQL query triggers a "large table without WHERE" warning
- **THEN** the frontend SHALL display a yellow banner above the result with the warning message

#### Scenario: Validation error displayed
- **WHEN** a SQL query is rejected by `isSelectQuery()` check
- **THEN** the frontend SHALL display a red banner with the error message and the Agent SHALL attempt to correct the query
